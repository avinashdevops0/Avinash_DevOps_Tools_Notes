# Jenkins Interview Questions & Answers

---

## 🟢 SECTION 1: FRESHER LEVEL (0–1 Year)

---

### Q1. What is Jenkins?
Jenkins is an open-source automation server written in Java. It is used to automate the parts of software development related to building, testing, and deploying, facilitating Continuous Integration (CI) and Continuous Delivery (CD).

---

### Q2. What is Continuous Integration (CI)?
CI is a development practice where developers frequently merge their code changes into a shared repository. Each merge triggers an automated build and test process, helping teams detect problems early.

---

### Q3. What is Continuous Delivery (CD)?
CD is the practice of automatically preparing code changes for release to production. Every successful build is ready to deploy, but deployment is done manually.

**Continuous Deployment** goes one step further — every passing build is automatically deployed to production without manual approval.

---

### Q4. What is a Jenkins Pipeline?
A Jenkins Pipeline is a suite of plugins that supports implementing and integrating continuous delivery pipelines into Jenkins. It defines the entire build/test/deploy process as code using a `Jenkinsfile`.

---

### Q5. What is a Jenkinsfile?
A `Jenkinsfile` is a text file that contains the definition of a Jenkins Pipeline and is checked into source control. It can be written in two syntaxes:
- **Declarative** — simpler, structured
- **Scripted** — more flexible, uses Groovy

---

### Q6. What are the types of Jenkins jobs?
- **Freestyle Project** — simple, GUI-based job
- **Pipeline** — code-based CI/CD pipeline
- **Multibranch Pipeline** — auto-creates pipelines for each branch in a repo
- **Folder** — organizes jobs into groups
- **Multi-configuration Project** — runs same job with different configurations

---

### Q7. What is a Jenkins Agent/Node?
An agent (or node) is a machine that executes the jobs assigned by the Jenkins master. It helps distribute the workload. The `master` handles scheduling; `agents` run the actual build steps.

---

### Q8. What is a Build Trigger in Jenkins?
A build trigger defines when a job should be executed automatically. Common triggers:
- **SCM Polling** — checks repo at regular intervals
- **Webhook** — triggered by a push event from GitHub/GitLab
- **Cron schedule** — triggered at a specific time (e.g., `H 2 * * *`)
- **Upstream job** — triggered when another job completes

---

### Q9. What is the difference between Declarative and Scripted Pipeline?

| Feature | Declarative | Scripted |
|---|---|---|
| Syntax | Structured, predefined | Groovy-based, flexible |
| Ease of use | Easier for beginners | Requires Groovy knowledge |
| Error handling | Built-in `post` block | Manual try-catch |
| Recommended for | Most use cases | Complex custom logic |

---

### Q10. What are Jenkins Plugins?
Plugins extend Jenkins functionality. Examples:
- **Git Plugin** — integrates with Git
- **Maven Plugin** — builds Java projects
- **Docker Plugin** — integrates Docker
- **Slack Notification Plugin** — sends build alerts to Slack
- **Blue Ocean** — modern UI for pipelines

---

### Q11. What is the Jenkins Master-Agent Architecture?
- **Master** — manages the Jenkins UI, schedules jobs, dispatches builds to agents, and monitors results
- **Agent** — connects to master and executes the builds
- Communication happens via SSH or JNLP protocol

---

### Q12. Where are Jenkins builds stored?
Jenkins stores build data in `$JENKINS_HOME` directory (usually `/var/jenkins_home` or `/var/lib/jenkins`). Each job has its own folder with workspace, logs, and artifacts.

---

### Q13. What is a Workspace in Jenkins?
A workspace is a directory on the agent where Jenkins checks out source code and runs build commands for a specific job.

---

### Q14. What is an Artifact in Jenkins?
An artifact is any file generated by a build (e.g., `.jar`, `.war`, `.zip`) that is archived and made available for download or use by downstream jobs.

---

### Q15. What is the use of `post` block in a Declarative Pipeline?
The `post` block defines actions to be taken after the pipeline or stage completes. Common conditions:
- `always` — runs regardless of result
- `success` — runs only if build succeeds
- `failure` — runs only if build fails
- `unstable` — runs if build is unstable
- `changed` — runs if result changed from previous build

## 🟡 SECTION 2: INTERMEDIATE LEVEL (1–3 Years)

---

### Q16. Explain a basic Declarative Pipeline syntax.

```groovy
pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                sh 'mvn clean package'
            }
        }
        stage('Test') {
            steps {
                sh 'mvn test'
            }
        }
        stage('Deploy') {
            steps {
                sh './deploy.sh'
            }
        }
    }
    post {
        failure {
            mail to: 'team@example.com', subject: 'Build Failed'
        }
    }
}
```

---

### Q17. What is a Multibranch Pipeline? When do you use it?
A Multibranch Pipeline automatically discovers, manages, and executes Pipelines for branches and PRs in a repository. Jenkins scans the repo and creates a pipeline for each branch that contains a `Jenkinsfile`. Use it when you have multiple feature branches and want isolated CI for each.

---

### Q18. How do you pass parameters to a Jenkins job?

```groovy
pipeline {
    agent any
    parameters {
        string(name: 'ENV', defaultValue: 'dev', description: 'Target environment')
        booleanParam(name: 'RUN_TESTS', defaultValue: true, description: 'Run tests?')
        choice(name: 'REGION', choices: ['us-east-1', 'eu-west-1'], description: 'AWS Region')
    }
    stages {
        stage('Deploy') {
            steps {
                echo "Deploying to ${params.ENV} in ${params.REGION}"
            }
        }
    }
}
```

---

### Q19. How do you use environment variables in Jenkins?

```groovy
pipeline {
    agent any
    environment {
        APP_NAME = 'myapp'
        BUILD_ENV = 'production'
    }
    stages {
        stage('Info') {
            steps {
                echo "Building ${env.APP_NAME} for ${env.BUILD_ENV}"
                echo "Jenkins built #${env.BUILD_NUMBER}"
            }
        }
    }
}
```

Built-in variables: `BUILD_NUMBER`, `BUILD_URL`, `JOB_NAME`, `WORKSPACE`, `GIT_BRANCH`, `GIT_COMMIT`

---

### Q20. How do you handle credentials in Jenkins securely?
Use Jenkins Credentials Store. Never hardcode passwords in Jenkinsfile.

```groovy
stage('Deploy') {
    steps {
        withCredentials([usernamePassword(
            credentialsId: 'my-db-creds',
            usernameVariable: 'DB_USER',
            passwordVariable: 'DB_PASS'
        )]) {
            sh 'deploy.sh $DB_USER $DB_PASS'
        }
    }
}
```

Types of credentials: Username/Password, Secret Text, SSH Key, Certificate, AWS credentials

### Q21. How do you run parallel stages in Jenkins?

```groovy
stage('Parallel Tests') {
    parallel {
        stage('Unit Tests') {
            steps { sh 'mvn test -Dtest=UnitTests' }
        }
        stage('Integration Tests') {
            steps { sh 'mvn test -Dtest=IntegrationTests' }
        }
        stage('Security Scan') {
            steps { sh 'run-security-scan.sh' }
        }
    }
}
```

### Q22. What is a Shared Library in Jenkins?
A Shared Library is a way to share reusable pipeline code across multiple Jenkinsfiles. It's stored in a separate Git repo and structured as:

```
(root)
├── vars/
│   └── deployApp.groovy     ← global functions
├── src/
│   └── org/example/Utils.groovy  ← helper classes
└── resources/               ← static files
```

Usage in Jenkinsfile:
```groovy
@Library('my-shared-library') _
deployApp('myservice', 'production')
```

### Q23. How do you configure Jenkins to send email notifications?

```groovy
post {
    failure {
        emailext(
            subject: "FAILED: ${env.JOB_NAME} #${env.BUILD_NUMBER}",
            body: "Check console: ${env.BUILD_URL}",
            to: 'devteam@example.com'
        )
    }
}
```

Requires Email Extension Plugin and SMTP configuration in `Manage Jenkins → Configure System`.

### Q24. What is Blue Ocean in Jenkins?
Blue Ocean is a modern, visual UI for Jenkins Pipelines. It provides:
- Visual pipeline editor
- Clear visualization of parallel stages
- Integrated pipeline creation from SCM
- Better error highlighting

### Q25. How does Jenkins integrate with Docker?

```groovy
pipeline {
    agent {
        docker {
            image 'maven:3.8.6-jdk-11'
            args '-v /root/.m2:/root/.m2'
        }
    }
    stages {
        stage('Build') {
            steps {
                sh 'mvn clean package'
            }
        }
    }
}
```

You can also build and push Docker images:
```groovy
steps {
    script {
        docker.build("myapp:${env.BUILD_NUMBER}")
              .push("latest")
    }
}
```

### Q26. What is the difference between `input` step and `approval` in Jenkins?
The `input` step pauses the pipeline and waits for human approval before proceeding. Often used before production deployments.

```groovy
stage('Deploy to Production') {
    steps {
        input message: 'Approve deployment to PROD?', ok: 'Deploy'
        sh './deploy-prod.sh'
    }
}
```

---

### Q27. How do you archive artifacts and publish test results?

```groovy
post {
    always {
        archiveArtifacts artifacts: 'target/*.jar', fingerprint: true
        junit 'target/surefire-reports/*.xml'
        publishHTML([
            reportDir: 'target/site',
            reportFiles: 'index.html',
            reportName: 'Test Report'
        ])
    }
}
```

---

### Q28. What is Jenkins Job DSL?
Job DSL is a plugin that allows you to define Jenkins jobs as code using a Groovy-based DSL. Unlike Jenkinsfile (which defines the pipeline logic), Job DSL defines the job configuration itself — triggers, SCM settings, parameters, etc.

---

### Q29. How do you back up Jenkins?
- Back up `$JENKINS_HOME` directory (contains all job configs, plugins, credentials)
- Use the **ThinBackup** plugin for scheduled backups
- Store Jenkins config as code using **JCasC (Jenkins Configuration as Code)** plugin
- Use SCM to version control Jenkinsfiles and Job DSL scripts

---

### Q30. What is Jenkins Configuration as Code (JCasC)?
JCasC allows you to define Jenkins system configuration in a YAML file, which can be version-controlled. Example:

```yaml
jenkins:
  systemMessage: "Jenkins managed by JCasC"
  numExecutors: 5
  agentProtocols:
    - "JNLP4-connect"
credentials:
  system:
    domainCredentials:
      - credentials:
          - usernamePassword:
              id: "github-creds"
              username: "user"
              password: "${GITHUB_TOKEN}"
```

## 🔴 SECTION 3: SCENARIO-BASED QUESTIONS

---

### Scenario 1: Your pipeline fails intermittently. How do you debug it?

**Answer:**
1. Check the **Console Output** for exact error messages
2. Look for flaky tests — re-run the failing stage in isolation
3. Add `echo` or `sh 'env'` statements to print environment variables
4. Check **agent availability** — maybe the node went offline
5. Review **race conditions** in parallel stages
6. Use `timestamps()` wrapper to track timing issues:
   ```groovy
   options { timestamps() }
   ```
7. Enable **verbose logging** for shell steps:
   ```groovy
   sh 'set -x; ./build.sh'
   ```

---

### Scenario 2: You need different deployment steps for dev, staging, and prod. How do you handle it?

**Answer:** Use parameters or branch-based conditions:

```groovy
pipeline {
    agent any
    parameters {
        choice(name: 'DEPLOY_ENV', choices: ['dev', 'staging', 'prod'])
    }
    stages {
        stage('Deploy') {
            steps {
                script {
                    if (params.DEPLOY_ENV == 'prod') {
                        input message: 'Approve PROD deployment?'
                        sh './deploy.sh prod'
                    } else {
                        sh "./deploy.sh ${params.DEPLOY_ENV}"
                    }
                }
            }
        }
    }
}
```

---

### Scenario 3: A developer pushes to the main branch and wants to trigger automatic deployment. How do you set this up?

**Answer:**
1. Configure a **GitHub/GitLab Webhook** pointing to `http://<jenkins-url>/github-webhook/`
2. In Jenkins job, set trigger:
   ```groovy
   triggers {
       githubPush()
   }
   ```
3. Add branch condition:
   ```groovy
   when {
       branch 'main'
   }
   ```
4. Full flow: Push → Webhook → Jenkins triggers → Build → Test → Deploy

---

### Scenario 4: Your Jenkins build is very slow. How do you optimize it?

**Answer:**
1. **Run stages in parallel** — split test suites into parallel stages
2. **Use faster agents** — more CPU/RAM or use cloud-based ephemeral agents
3. **Cache dependencies** — mount Maven/npm cache as Docker volume
4. **Avoid polling** — use webhooks instead of SCM polling
5. **Limit workspace cleanup** — only clean when necessary
6. **Use incremental builds** — skip unchanged modules
7. **Distribute load** — add more agents to the Jenkins pool

---

### Scenario 5: You need to deploy to 5 different servers simultaneously. How do you do it?

**Answer:**

```groovy
stage('Deploy to All Servers') {
    parallel {
        stage('Server 1') {
            steps { sh 'ssh user@server1 ./deploy.sh' }
        }
        stage('Server 2') {
            steps { sh 'ssh user@server2 ./deploy.sh' }
        }
        stage('Server 3') {
            steps { sh 'ssh user@server3 ./deploy.sh' }
        }
    }
}
```

Or use a loop:
```groovy
def servers = ['server1', 'server2', 'server3', 'server4', 'server5']
def deployStages = servers.collectEntries { server ->
    ["Deploy to ${server}": {
        node {
            sh "ssh user@${server} ./deploy.sh"
        }
    }]
}
parallel deployStages
```

### Scenario 6: How do you ensure secrets are never printed in Jenkins logs?

**Answer:**
1. Always use `withCredentials` block — Jenkins masks values automatically
2. Never use `echo $PASSWORD` — use masked variables only in commands
3. Use `set +x` to disable shell tracing before sensitive commands:
   ```groovy
   sh '''
       set +x
       curl -u $USER:$PASS https://api.example.com
   '''
   ```
4. Use **HashiCorp Vault Plugin** for enterprise secret management
5. Audit logs regularly with the **Audit Trail** plugin

---

### Scenario 7: A job runs every night and you want to skip it on weekends. How?

**Answer:**

```groovy
triggers {
    cron('H 2 * * 1-5')  // Mon-Fri at ~2am only
}
```

Or conditionally skip:
```groovy
stage('Nightly Job') {
    when {
        expression {
            def day = new Date().format('EEE')
            return !(day in ['Sat', 'Sun'])
        }
    }
    steps {
        sh './nightly-task.sh'
    }
}
```

---

### Scenario 8: How do you roll back a failed deployment in Jenkins?

**Answer:**
1. **Keep the previous build artifact** using `archiveArtifacts`
2. Create a **rollback job** that copies previous artifact and redeploys
3. Use **parameterized build** — accept a build number and redeploy that artifact:
   ```groovy
   parameters {
       string(name: 'ROLLBACK_BUILD', description: 'Build # to rollback to')
   }
   ```
4. In Kubernetes setups, use `kubectl rollout undo deployment/myapp`
5. In Docker/Docker Swarm, tag images by build number and re-deploy the old tag

---

### Scenario 9: Jenkins master goes down. What's your recovery plan?

**Answer:**
1. Jenkins config is stored in `$JENKINS_HOME` — restore from backup
2. If using **JCasC**, re-apply the YAML config to a new instance
3. **Pipelines continue running on agents** even if master restarts (in some configurations)
4. Set up **Jenkins HA** with a standby master or use Kubernetes-based Jenkins (Jenkins Operator)
5. Use **S3/NFS** for shared `$JENKINS_HOME` across instances
6. Document recovery steps and test DR drills regularly

---

### Scenario 10: How do you manage Jenkins for a team of 50+ developers?

**Answer:**
1. **Role-Based Access Control (RBAC)** — use the Role Strategy Plugin to restrict permissions per team
2. **Folders** — organize jobs by team/project
3. **Shared Libraries** — enforce pipeline standards across teams
4. **JCasC** — manage Jenkins config as code for reproducibility
5. **Ephemeral agents** — use Kubernetes or EC2 agents that spin up on demand
6. **Separate instances** — consider having dev/prod Jenkins instances
7. **Audit Trail Plugin** — track who did what
8. **Resource Quotas** — limit concurrent builds per team
